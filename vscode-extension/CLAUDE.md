# Algorithm Visualizer — VS Code Extension

## What this project is

A VS Code / Cursor extension that lets developers select Python code, trigger a command, and get an AI-generated interactive step-by-step visualizer in a docked side panel. As the user steps through frames in the visualization, the corresponding source line is highlighted in the editor in real time (bidirectional sync).

The visualization is generated by calling the Claude API with a detailed prompt that instructs it to produce a complete self-contained HTML file — code viewer on the left with a line cursor, animated data-structure visuals on the right, and Prev/Next/Autoplay controls at the bottom.

## Architecture

```
src/
├── extension.ts        Entry point. Registers two commands and shows a
│                       one-time welcome notification.
│
├── VisualizerPanel.ts  Core class. Manages the singleton WebviewPanel,
│                       calls the Claude API, injects the VS Code bridge
│                       script, handles editor decoration sync, and disposes
│                       resources on panel close.
│
└── promptBuilder.ts    Single exported function buildPrompt(code, hint).
                        This is the most important file — the quality of
                        generated visualizations lives here.
```

### How bidirectional sync works

1. `promptBuilder.ts` instructs Claude to call `window.__algoVizNotify(lineNumber)` after every `render()` call.
2. `VisualizerPanel._injectBridge()` appends a `<script>` before `</body>` in every generated HTML page. That script acquires the VS Code API and wires `window.__algoVizNotify` to `vscode.postMessage({ type: 'algoVizFrame', line })`.
3. `VisualizerPanel` listens for those messages and applies a `TextEditorDecorationType` to the matching line in the source editor.

### Why the webview is replaced (not updated) on regenerate

VS Code doesn't offer a way to hot-swap webview content without a visible repaint. Since the generated HTML is a full self-contained document, the cleanest approach is setting `panel.webview.html` to the new content. `retainContextWhenHidden: true` preserves the current frame and autoplay state when the user switches tabs.

## Build & run

```bash
npm install          # install devDependencies (TypeScript, @types/vscode, vsce)
npm run compile      # tsc → out/
npm run watch        # tsc in watch mode (use alongside F5)
npm run package      # vsce package → algo-visualizer-0.1.0.vsix
```

Press **F5** in VS Code to launch the Extension Development Host with the extension loaded.

## Settings (package.json contributes.configuration)

| Key | Default | Purpose |
|-----|---------|---------|
| `algoViz.apiKey` | `""` | Anthropic API key — required |
| `algoViz.model` | `claude-sonnet-4-6` | Model for generation |
| `algoViz.syncEditorCursor` | `true` | Enable/disable line decoration sync |
| `algoViz.autoVisualize` | `true` | Generate immediately on panel open |

| Command ID | Title | Trigger |

| `algoViz.visualize` | Visualize Algorithm | Right-click, palette, `Cmd+Shift+Alt+V` |
| `algoViz.visualizeWithHint` | Visualize Algorithm with Hint… | Right-click, palette |

## Key invariants to preserve

- **`promptBuilder.ts` is the source of truth for generation quality.** When improving output, edit the prompt there — do not hard-code HTML in the TypeScript.
- **One panel at a time.** `VisualizerPanel.currentPanel` is a static singleton. Re-triggering the command reuses it.
- **No external bundler.** The extension uses plain `tsc` output (`commonjs`, `ES2020`). Keep it that way — no webpack or esbuild unless the extension grows significantly.
- **Bridge script must survive HTML variations.** `_injectBridge` inserts before `</body>` if present, or appends to the end. Keep this fallback.

## Extending the prompt

The prompt in `promptBuilder.ts` controls the visual language of every generated visualization. Things you can tune:

- **Color semantics**: `--accent` = primary cursor, `--info` = window/range, `--danger` = duplicate/error. If you want pointer colors specified more precisely, add an explicit mapping in the prompt.
- **Frame density**: The prompt doesn't specify a minimum frame count. Add "capture at least one frame per loop iteration body" if you want finer granularity.
- **New data structures**: Add bullet points to the "render data structures graphically" section for graphs (adjacency lists), trees (box-and-arrow), heaps, etc.
- **`window.__algoVizNotify`**: If you change the bridge function name, update both `_injectBridge()` and the prompt simultaneously.

## Common tasks for Claude Code

- **Improve visualization quality** → edit `src/promptBuilder.ts`, specifically the "LAYOUT", "FRAME MODEL", and "CONSTRAINTS" sections.
- **Add a new trigger** → add a command entry in `package.json` under `contributes.commands` and `contributes.menus`, then register it in `extension.ts`.
- **Change decoration style** → edit the `createTextEditorDecorationType` call in `VisualizerPanel` constructor.
- **Support other languages** → generalize `buildPrompt` to accept a `language` param and adjust the code fence and syntax guidance in the prompt.
- **Add streaming** → replace the single-shot `https.request` in `_callClaude` with a streaming request; send partial HTML chunks to the webview via `webview.postMessage` and append them progressively.
- **Build the IntelliJ plugin** → see the architecture notes in README.md. Key APIs: `ToolWindowFactory`, `JBCefBrowser`, `AnAction`, `RangeHighlighter`.
